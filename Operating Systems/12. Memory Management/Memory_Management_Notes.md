# Memory Management - Comprehensive Notes

## 1. Why Memory Management is Needed

### Key Decisions OS Must Make:
- **Which process gets memory?** — Allocation among multiple running processes
- **How much memory each process needs?** — Based on its workload requirements
- **When to place a process in memory?** — Timing of allocation
- **How to protect process memory?** — Prevent one process from accessing another's data (e.g., Chrome shouldn't access WhatsApp memory)
- **How to reduce memory overhead?** — Efficient usage of limited RAM

### OS Memory Management Functions:
- **Allocation** — Assigning memory to processes
- **Protection** — Isolating process addresses
- **Sharing** — Allowing controlled shared access
- **Tracking** — Monitoring free and used memory
- **Efficiency** — Maximizing resource utilization

---

## 2. Logical Address vs Physical Address
| Aspect | Logical (Virtual) Address | Physical Address |
|--------|---------------------------|------------------|
| **Source** | Generated by the CPU from the perspective of a running program | Address routed over physical traces into a specific cell inside the RAM chip |
| **Starts from** | Always zero (process sees a private, continuous block starting at index 0) | Can start at any location |
| **Visibility** | Hidden from program — program doesn't know its real location | Used directly by RAM |
| **Purpose** | Memory protection & abstraction | Real data access |

### Address Translation
- Performed by the **Memory Management Unit (MMU)** — a dedicated hardware component that maps logical addresses to physical addresses in real-time at run-time
- Converts: `Logical Address → Physical Address`
- MMU also checks if a process is accessing memory outside its allowed scope

---

## 3. Valid / Invalid Memory Access

A process should only access its own memory space.

### Check Mechanism:
```
Valid range = [base_address, base_address + limit)
```

| Condition | Result | Action |
|-----------|--------|--------|
| `logical_address < limit` | **Valid** | Allow access |
| `logical_address >= limit` | **Invalid** | Block access (OS intervenes) |

### Example:
- Base = 5000, Limit = 1000
- Logical address = 700 → 700 < 1000 → **Valid** → Can access
- Logical address = 1200 → 1200 >= 1000 → **Invalid** → Blocked
- Base = 12,000, Limit = 4,096
  - Valid addresses: 0 to 4,095 (never includes limit itself)
  - Logical address = 4,096 → **Invalid**

## 4. Contiguous Memory Allocation & Fragmentation

Contiguous memory allocation loads each program into a single, continuous block of physical memory. It manifests in two main strategies:

### A. Fixed (Static) Partitioning
- RAM is divided into **fixed-size, permanent partitions** at system boot
- Each process is loaded into one fixed partition
- **Drawback: Internal Fragmentation** — if process size < assigned partition size, the remaining unused space inside that partition is wasted and cannot be used by other processes

### B. Variable (Dynamic) Partitioning
- Partitions are created **dynamically based on the exact size required** by an incoming program
- **Drawback: External Fragmentation** — as programs load and terminate over time, free memory breaks up into tiny scattered holes. Total free memory may exceed the request, but no single contiguous hole is large enough to accommodate it
- **The Fix: Compaction** — OS shifts all allocated blocks to one side of RAM to join all tiny holes into one large continuous free block. However, compaction is **computationally expensive and slow**

### How Paging Solves the Problem
- Eliminates **external fragmentation** by allocating pages to any available frames (non-contiguous)
- Internal fragmentation remains only in the last frame (partial page waste)

---

## 5. Page Table & Address Translation

### Page Table:
- Maps each **page number → frame number**
- Each process has its own page table
- Used for faster physical address lookup

### Address Translation Steps:
1. Calculate **page number** and **offset** from logical address
2. Look up **frame number** in page table using page number
3. Compute **physical address**:

$$physical\_address = (frame\_number \times page\_size) + offset$$

### Example:
- Page size = 100 bytes, Logical address = 250
- Page number = 250 ÷ 100 = **2**
- Offset = 250 mod 100 = **50**
- Page table says: page 2 → frame 7
- Physical address = (7 × 100) + 50 = **750**

---

## 6. TLB (Translation Look-Aside Buffer)

### Problem with Paging Alone:
Every memory access requires going to the page table in RAM → **two-step process** → slow

### Solution: TLB
- A small, fast cache storing recently used page-to-frame mappings
- **TLB Hit**: Page number found in TLB → direct lookup (fast)
- **TLB Miss**: Page number not found → must access page table in RAM (slow)

### Effective Access Time (EAT):
```
hit_time = TLB_lookup_time + memory_access_time
miss_time = hit_time + page_table_access_time (≈ equals memory_access_time)
miss_ratio = 1 - hit_ratio

EAT = (hit_ratio × hit_time) + (miss_ratio × miss_time)
```

### Example:
- TLB lookup = 10 ns, Memory access = 100 ns, Hit ratio = 80%
- Hit time = 10 + 100 = **110 ns**
- Miss time = 110 + 100 = **210 ns**
- Miss ratio = 1 - 0.8 = **0.2**
- EAT = (0.8 × 110) + (0.2 × 210) = 88 + 42 = **130 ns**

## 7. Segmentation

Unlike paging (equal-sized blocks), segmentation divides a program into **logical, variable-sized units** based on human-centric or compiler-centric structural logic:

| Segment | Contents |
|---------|----------|
| **Code segment** | Program instructions |
| **Data segment** | Stored variables |
| **Stack segment** | Function calls, local variables |
| **Heap segment** | Dynamic memory allocation |

- Segments can have **different sizes** (variable-sized logical units)
- Paging is the primary mechanism in modern systems

---

## 8. Virtual Memory

### Core Concept:
- Not all pages of a process need to be loaded into RAM at once
- Only required pages are loaded; rest stays in secondary memory (disk)
- Each process gets its own **virtual address space** — feels like having private, fully available memory

### Valid Bit in Page Table:
| Valid Bit | Meaning | Terminology | Action |
|-----------|---------|-------------|--------|
| **1** | Page is in RAM | **Page Hit / Page Present** | Direct access |
| **0** | Page is in secondary memory | **Page Miss / Page Fault** (used interchangeably) | Fetch from disk via demand paging |

---

## 8a. Swapping

- A process can be **swapped temporarily out of memory** to a backing store, then later brought back into memory for continued execution
- The combined time for all swaps a process undergoes is called the **swap time**
- Critical concept in virtual memory — allows running processes whose sizes exceed physical RAM
- When brought back: must be loaded into **a different frame** if original frames no longer available

---

## 8b. Dynamic Loading

- A routine is **not loaded into memory until it is needed**
- Loaded from disk into an **unused, empty frame** when a process first calls that routine
- Does not increase overall system performance; primarily beneficial for I/O-heavy programs running on small memory configurations
- Well-structured programs using dynamic loading can run on machines with less physical RAM than would otherwise be required

---

## 8c. Memory-Mapped Files (mmap)

- Treats **files in secondary storage as extensions of main memory**
- A file mapping (a chunk of bytes on disk) is mapped into a process's address space
- Once mapped, reads/writes to that region access the file directly — **no explicit read/write system calls** needed
- Eliminates I/O system calls for certain workflows but introduces coupling between file contents and memory
- Commonly used for:
  - Shared libraries (.dll, .so) loaded into multiple processes
  - Large data files accessed randomly rather than sequentially

---

## 9. Demand Paging & Page Faults

### Demand Paging:
Loading a page into memory **only when it is needed**

### Page Fault Workflow:
1. **Trap to OS** — valid bit = 0 (not in RAM) triggers a page fault interrupt → control transfers to the OS
2. **Locate page on disk** — OS finds the missing page block in secondary storage
3. **Find free frame** — OS locates an empty physical RAM frame slot
4. **Read page into frame** — OS reads the data block from disk into the designated RAM frame
5. **Update page table** — OS updates the page table entry with the correct frame mapping and sets its valid bit to 1
6. **Restart instruction** — CPU restarts the exact instruction that triggered the page fault

### If RAM is completely full during a page fault:
- The OS must use a page replacement algorithm to select a victim page, write it back to disk if modified (dirty), mark its page table entry as invalid, and overwrite that frame with the incoming page

### Page Fault Rate:
```
page_fault_rate = total_page_faults / total_page_references
```

## 10. Page Replacement Algorithms

Used when RAM is full and a new page must be loaded.

### A. FIFO (First In, First Out)
- Evict the **oldest** page (first entered)
- Behaves like a queue
- Simplest but not optimal

#### FIFO Example: Frames = 3, Reference string = 1, 2, 3, 1, 4, 2
```
Frame state evolution:
[1]       → miss
[1,2]     → miss
[1,2,3]   → miss
[1,2,3]   → hit (1 already present)
[4,2,3]   → miss (evict 1 — oldest)
[4,2,3]   → hit (2 already present)

Hits = 2, Misses/Page faults = 4
```

### B. Optimal Page Replacement
- Evict the page that will **not be used for the longest time** in the future
- Guarantees minimum page faults
- **Cannot be physically implemented** — requires non-deterministic, perfect knowledge of future execution events
- Used as a benchmark only

#### Example: Frames = 3, Reference string = 7, 0, 1, 2, 0, 3, 0, 4, 2, 3
- Page faults = **6** (remove pages not needed in future)

### C. LRU (Least Recently Used)
- Evict the page that has **not been used for the longest time** in the past
- Predicts future by observing past usage
- Practical and widely used

#### Example: Frames = 3, Reference string = 1, 2, 4, 1, 2, 5, 1, 2, 3, 4, 5
- Evict the least recently seen page at each step
- Page faults = **10**

---

## 11. Belady's Anomaly

- Sometimes **increasing frame size increases page faults**
- Occurs specifically with **FIFO** algorithm
- Does NOT occur with Optimal or LRU (more frames always help)

### Why in FIFO:
Adding more frames can change eviction order such that previously cached pages are evicted earlier, causing more faults.

---

## 12. Thrashing

- **Definition**: A critical failure state where the OS spends more time swapping pages in and out than executing instructions — throughput drops to near-zero
- **Cause**: The combined working memory footprints (working sets) of active processes drastically exceed the available physical RAM capacity

### Effects:
- Page fault rate skyrockets
- CPU utilization drops significantly (overhead of fetching from disk)
- System performance severely degraded

## 13. What Is Inside a Program (Process Structure)

When a program runs, it becomes a **process** which contains:
- **Code (text segment)** — Program instructions
- **Variables (data segment)** — Stored values
- **Objects** — Runtime objects for OOP languages
- **Stack** — Function calls, return addresses, local variables
- **Heap** — Dynamically allocated memory
- **Temporary data** — Intermediate values during execution

## 14. Page Table Entries (PTE) Format

Each entry in the page table contains:

| Field | Description |
|-------|-------------|
| **Valid bit** | 1 = page is in RAM; 0 = page is in secondary memory (not loaded yet) |
| **Frame number** | The physical frame where the page is stored |
| **Optional bits** | Protection bits, dirty bit, access bit, etc. |

### Page Table Walk:
```
1. CPU generates logical address
2. MMU splits it into [page number | offset]
3. Lookup page number in page table → get frame number (if valid bit = 1)
4. Physical address = (frame_number × page_size) + offset
5. If valid bit = 0 → **Page Fault** occurs
```

---

## 15. TLB Mapping & Invalidation

### When TLB is Updated:
- On every memory access, the mapping is cached in TLB
- TLB is smaller and faster than main memory (implemented in hardware)

### Cache Coherence Problem:
After a page replacement occurs, the OS must **invalidate** stale TLB entries for that process.

### TLB Shootdown:
When switching between processes:
1. **TLB flush** — Clear all TLB entries before context switch
2. Each new process builds fresh TLB as it accesses memory (fill-up pattern)

---

## 16. Context Switch in Paging Systems

| Step | Action |
|------|--------|
| 1 | Save current process's page table pointer (PTBR — Page Table Base Register) |
| 2 | Load new process's page table pointer into PTBR |
| 3 | **Flush TLB** — All old mappings are invalid |
| 4 | New process starts executing; TLB fills up gradually |

---

## 17. Segmentation with Paging (Segmented Paging)

### Pure Segmentation Problem:
- Suffers from **external fragmentation**
- Requires contiguous segments in memory

### Solution: Segmented Paging
Combine segmentation and paging:
1. Divide program into **segments** (logical units)
2. Each segment is divided into **pages**
3. Each segment has its own page table
4. Eliminates external fragmentation within segments

## Quick Reference Formulas

| Concept | Formula |
|---------|---------|
| Number of pages | `ceil(process_size / page_size)` |
| Number of frames | `physical_memory_size / frame_size` |
| Page number | `logical_address ÷ page_size` (integer division) |
| Offset | `logical_address mod page_size` (remainder) — defines the exact position of data within a specific page; this value passes through address translation unchanged |
| Physical address | `(frame_number × page_size) + offset` |
| Internal fragmentation | `allocated_memory - process_size` |
| Hit time | `TLB_lookup_time + memory_access_time` |
| Miss time | `hit_time + page_table_access_time` |
| EAT | `(hit_ratio × hit_time) + (miss_ratio × miss_time)` |
| Page fault rate | `total_page_faults / total_page_references` |
